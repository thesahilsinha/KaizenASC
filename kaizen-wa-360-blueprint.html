// 01 — What this is

The Product

Kaizen WA 360 is a white-label WhatsApp automation SaaS. The builder (Sahil / Kaizen ASC) runs one deployment. Each paying client gets their own login, their own WhatsApp number connected, their own database, and their own branded experience.

📱
QR Connect Once
Client scans QR code once. Session is saved to their Supabase. Auto-reconnects on engine restart. No re-scanning.
🤖
Chatbot Flows
Visual if/else keyword rule builder. Sequence flows (reply 1 for X, 2 for Y). Fallback reply. All stored as JSON in Supabase.
📣
Bulk Campaigns
Upload CSV contacts. Set message template. Schedule time. Engine sends with random delays (anti-ban). Daily cap respected.
💬
Live Inbox
All incoming messages logged to Supabase in real time. Client sees full conversation history per contact in their dashboard.
👥
Contact Manager
Import via CSV. Tag contacts (lead, customer, VIP). Filter by tag for targeted campaigns. View message history per contact.
📖
User Guide
Built-in collapsible user guide inside every client dashboard — same pattern as Seven Elements Mailer. No external docs needed.
👑
Super Admin Panel
Only Sahil sees this. Add clients, store their Supabase keys, track payments, renewals, session health. Full client CRM.
💰
Flexible Billing
No fixed pricing in code. Setup fee + monthly retainer set per client while onboarding. Renewal dates tracked in admin.

// 05 — Chatbot logic

How the Chatbot Works

Every incoming WhatsApp message fires a messages.upsert event in Baileys. The engine fetches the client's flow rules from their Supabase and replies accordingly.

Message flow
1
Message arrives on client's WA number
Baileys socket fires event. Engine captures: sender phone, message text, clientId.
2
Engine fetches chatbot_flows from client's Supabase
Ordered by priority ASC. Checks each rule in order.
3
Rule matching
For each active rule: does message contain / equal / start with trigger_keyword? First match wins.
4
Send reply via Baileys
socket.sendMessage(senderPhone, { text: rule.reply }). Adds typing indicator simulation first.
5
Log to messages_log
Both inbound message and outbound reply saved to client's Supabase. Client sees it in Conversations tab live.
✦
No match → default fallback rule
priority 999 rule always exists. Client sets their fallback reply in flow builder.
Example rules (as client sees them in builder)
IF contains price → reply Our plans start at ₹999/month. DM for details!
IF contains demo → reply Book your free demo here: calendly.com/xyz
IF contains hi / hello / hey → reply Welcome to XYZ! Reply 1 for pricing, 2 for demo, 3 for support
DEFAULT fallback → reply Thanks for reaching out! Our team will get back to you shortly.
⚠️
Anti-ban rules baked into engine: Random delay 8–15s between outbound messages. Daily cap of 300 messages per number. Typing indicator before every reply. Sessions warmed up gradually on new numbers. Never send to opted_out contacts.

// 08 — Build order

How to Build This

Build in this exact order. Each step unlocks the next. Don't jump ahead.

1
SQL Template Script
Write the full .sql file with all tables, RLS policies, and a default chatbot fallback rule. Run once per client. Test on a blank Supabase project.
2
Master Supabase (yours)
Create your own Supabase. Create master_clients table. Add yourself as superadmin user with role metadata.
3
Engine — Baileys core (wa-engine)
Node.js + Express. One endpoint: POST /connect/:clientId. Starts Baileys socket, streams QR via Socket.io, saves session to client Supabase on connect. Test locally first.
4
Engine — Chatbot handler
On messages.upsert: fetch flows from client Supabase, match rules, reply, log to messages_log.
5
Engine — Campaign sender
POST /campaign/send/:clientId. Fetch contacts by tag, queue messages with random delays, respect daily_limit, update sent_count.
6
Deploy engine to Fly.io
Write Dockerfile. fly launch. fly deploy. Set env vars. Test QR + chatbot on live URL.
7
Next.js — Auth + routing (wa-dashboard)
Supabase auth. Middleware to check role. /admin/* for superadmin, /dashboard/* for clients. Dynamic Supabase client init per session using stored keys.
8
Next.js — Client pages
Dashboard, Contacts, Campaigns, Chatbot builder, Conversations inbox, WA Connect (QR page), User Guide, Settings.
9
Next.js — Admin pages
Client list, Add client form (with billing fields), Payments view, Session health monitor.
✦
Deploy to Vercel. Onboard first client.
Add client in admin → create their Supabase → run SQL script → give login → they scan QR → live in 30 mins.
